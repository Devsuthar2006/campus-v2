# Phase 08 — Notification System

> **Milestone:** Users receive timely, respectful in-app (and email) notifications across features.

## Objective
Implement the multi-channel notification system: in-app (realtime + persisted), email via a queue, and future push. It connects the events generated by Phases 01–07 (friend requests, matches, messages, wall activity) into re-engagement, with per-user preferences and batching that respect attention. This is a cross-cutting phase that retro-wires earlier features' notifications.

## Features Included
- In-app notifications (realtime delivery + durable history).
- Email notifications via queue; notification preferences (per-category/channel).
- Read/mark-all-read/dismiss; batching of low-priority signals; priority levels.

## Dependencies
- **Documentation:** `NOTIFICATION_SYSTEM.md` (all), `DATABASE_SCHEMA.md` §16 (Notifications module), `SOCKET_EVENTS.md` §10, `API_SPEC.md` §13.
- **Phases:** 00–07 (events to notify on). Email provider decision required (REVIEW_REPORT M-4).

## Backend Tasks
- Implement the notification service (`ARCHITECTURE.md` §2.11) over `DATABASE_SCHEMA.md` §16: `notifications`, `notification_preferences`, `notification_queue`.
- Implement triggers from existing domain events (friend request/accept, match, message, wall reply/reaction, announcements) — persist-then-deliver.
- Implement preference + quiet-hours checks, de-duplication, batching of low-priority (reactions) notifications, and priority handling (`NOTIFICATION_SYSTEM.md` §8, §10).
- Implement the email channel via `notification_queue` + chosen provider (M-4); push reserved (future).
- Endpoints: list, mark read, mark-all-read, dismiss, get/update preferences (`API_SPEC.md` §13).

## Frontend Tasks
- Notification center (list, unread, mark read), badge counts on nav, preference controls in Settings, real-time updates via socket.

## Database Tasks
- Implement `DATABASE_SCHEMA.md` §16 tables; unread partial index; partition-ready `notifications` by `created_at` (REVIEW_REPORT H-2 composite PK note).

## Socket Tasks
- Implement `notification_created`, `notification_read`, `notification_removed` to the user room (`SOCKET_EVENTS.md` §10). Realtime for online; persisted for offline.

## UI Components
- Notification center, notification item, unread badge, preference toggles, empty state ("You're all caught up") — per `UI_GUIDELINES.md` §16. Reaction notifications batched.

## Security Considerations
- Notifications never leak private content inappropriately; respect privacy/visibility; preferences honored; critical/security notifications cannot be fully muted (`SECURITY.md` §8; `NOTIFICATION_SYSTEM.md` §5).

## Acceptance Criteria
- Friend requests, matches, and messages generate timely in-app notifications; offline users see them on return; email fires per preference.
- Preferences and silent mode are respected; low-priority signals batch; critical notifications always reach the user.

## Deliverables
- Cross-cutting notification system driving relevant re-engagement across all built features.

## Out of Scope
- Push notifications, AI summaries/smart digest (future — `NOTIFICATION_SYSTEM.md` §11); notifications for not-yet-built features (added in their phases).

## Risks
- Over-notification (mitigated by batching + preferences); email-provider dependency (M-4). Multi-instance delivery needs the queue (already designed).

## Future Improvements
- Push, quiet hours, smart digest, cross-device read sync (`NOTIFICATION_SYSTEM.md` §11).
